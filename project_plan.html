แผนงานโครงการระบบ NDID ความเร็วสูงบนบล็อกเชน

Enterprise Hybrid Architecture with Rust, QUIC Auto-Switch & GraphQL Gateway

ประเภทระบบ: Decentralized Identity (NDID) สเปคหลัก: Rust (Memory Safety & High Concurrency) สถานะเอกสาร: แผนงานสถาปัตยกรรม (Architecture Blueprint)

1. บทนำและเป้าหมายระบบ (Project Overview)

โครงการนี้มุ่งเน้นการสร้างระบบ National Digital ID (NDID) ยุคใหม่เพื่อเชื่อมต่อระหว่างกลุ่มธนาคารพาณิชย์ โดยมีโจทย์หลักคือ "ต้องปลอดภัยสูงสุดตามกฎหมาย PDPA" และ "ต้องประมวลผลเร็วระดับเสี้ยววินาที (Real-time)" ด้วยการนำข้อดีของระบบ Centralized Database (ในธนาคาร) มาผสมผสานกับ Decentralized Ledger (Blockchain) แบบ Hybrid

ทำไมต้องใช้ Rust? Rust มีโครงสร้างพื้นฐานที่เด่นเรื่อง Memory Safety ปราศจากช่องโหว่ประเภท Buffer Overflow และมีประสิทธิภาพการทำงานที่สูงเทียบเท่าภาษา C/C++ เหมาะสำหรับการประมวลผลข้อมูลทางการเงินและการเข้ารหัสชั้นสูง (Cryptographic Operations)

3. แผนกลยุทธ์ความเร็วและความปลอดภัย (Speed & Security Strategy)

3.1 การรักษาความปลอดภัยและความเป็นส่วนตัว (Security & Privacy)

  • Zero Data on Chain: ห้ามนำข้อมูลระบุตัวตนบุคคล (PII) เช่น ชื่อ นามสกุล หรือรูปถ่าย ขึ้นสู่บล็อกเชนโดยเด็ดขาด บนบล็อกเชนจะเก็บเฉพาะ ค่า Hash (SHA-256) และสถานะธุรกรรมเท่านั้น
  • Encrypted P2P Data Tunnel: เมื่อเกิดการอนุมัติแชร์ข้อมูล ข้อมูลดิบจะถูกส่งตรงจากธนาคารต้นทาง (IdP) ไปยังธนาคารปลายทาง (RP) แบบ Peer-to-Peer โดยตรง ไม่ผ่านตัวกลางใดๆ
  • Permissioned Blockchain: เลือกใช้ Enterprise Blockchain (เช่น Substrate Framework) ที่จำกัดผู้มีสิทธิ์เข้าร่วมเฉพาะสถาบันการเงินที่ได้รับอนุญาตเท่านั้น

3.2 การประมวลผลความเร็วสูง (High Performance)

  • QUIC (HTTP/3)เป็นแกนหลัก: ใช้โปรโตคอล QUIC เพื่อลดขั้นตอน Handshake ในการเชื่อมต่อเครือข่ายระหว่างโหนดจากปกติลงเหลือเพียง 0-RTT หรือ 1-RTT ทำให้ตอบสนองเร็วกว่า TCP แบบดั้งเดิม
  • QUIC / TCP Auto-Switch (Fallback Mechanism): เพิ่มความทนทาน (Resilience) ให้กับระบบ หากเครือข่ายปลายทางหรือไฟร์วอลล์บล็อกพอร์ต UDP ระบบจะสลับมาใช้ท่อ TCP + TLS 1.3 ทันทีโดยออโต้
  • Asynchronous Core: พัฒนาสถาปัตยกรรมตัวเซิร์ฟเวอร์บนเครื่องยนต์ tokio (Asynchronous runtime ของ Rust) เพื่อรองรับ Concurrency ระดับหลายหมื่นธุรกรรมพร้อมกัน
ข้อควรระวังฝั่งเครือข่ายธนาคาร: เนื่องจาก QUIC ทำงานบน UDP ฝ่ายความปลอดภัยของธนาคารมักบล็อกพอร์ตประเภทนี้เป็นค่าเริ่มต้น แผนงานจึงระบุให้เปิดใช้ระบบ Auto-Switch บนฝั่ง Client สำหรับลูกค้าภายนอก แต่ฝั่งหลังบ้านระหว่างธนาคาร (Interbank) จะต้องล็อกพอร์ตและทำ Allowlist ไอพีร่วมกันอย่างเคร่งครัด

5. ตัวอย่างซอร์สโค้ดหลักของระบบ (Core Implementation Snippet)

ด้านล่างนี้คือโค้ดตัวอย่างในภาษา Rust ที่ผสานรวม GraphQL Entry Point เข้ากับระบบ QUIC/TCP Fallback Logic:

use async_graphql::{Object, Context, SimpleObject};
use tokio::time::{timeout, Duration};

// 1. โครงสร้างข้อมูลส่งกลับผ่าน GraphQL ไปยังหน้าบ้าน
#[derive(SimpleObject)]
pub struct IdentityStatus {
    pub request_id: String,
    pub status: String,          // Pending, Approved, Rejected
    pub active_protocol: String, // QUIC (0-RTT) หรือ TCP (Fallback)
}

pub struct QueryRoot;

#[Object]
impl QueryRoot {
    // 2. GraphQL Resolver สำหรับตรวจดูสถานะ NDID ความเร็วสูง
    async fn verify_ndid_record(&self, _ctx: &Context<'_>, request_id: String) -> IdentityStatus {
        let target_bank_node = "10.0.1.50:443"; // ไอพีปลายทางของธนาคารร่วมเครือข่าย
        
        // เรียกใช้ฟังก์ชันสลับท่อเครือข่ายอัจฉริยะ
        let protocol_used = match connect_with_fallback(target_bank_node).await {
            NetworkChannel::Quic => "QUIC (0-RTT) - Ultra Fast",
            NetworkChannel::Tcp  => "TCP + TLS 1.3 - Fallback Mode",
        };

        IdentityStatus {
            request_id,
            status: "Approved".to_string(),
            active_protocol: protocol_used.to_string(),
        }
    }
}

// 3. ระบบจำลองกลไก Auto-Switch (Happy Eyeballs Logic)
enum NetworkChannel { Quic, Tcp }

async fn connect_with_fallback(addr: &str) -> NetworkChannel {
    let quic_timeout = Duration::from_millis(500); // หาก QUIC โดนบล็อกหรือช้าเกิน 500ms ให้สลับทันที

    match timeout(quic_timeout, simulate_quic_connect(addr)).await {
        Ok(Ok(_)) => NetworkChannel::Quic, // สามารถเชื่อมต่อผ่าน UDP/QUIC ได้สำเร็จ
        _ => {
            // เกิด Timeout หรือเกิด Error (เช่น โดนบล็อกพอร์ต UDP) สลับไป TCP อัตโนมัติ
            simulate_tcp_connect(addr).await;
            NetworkChannel::Tcp
        }
    }
}

async fn simulate_quic_connect(_addr: &str) -> Result<(), ()> { Ok(()) }
async fn simulate_tcp_connect(_addr: &str) {}

6. ตารางเปรียบเทียบสถาปัตยกรรม (Comparison Matrix)

มิติการเปรียบเทียบ ระบบดั้งเดิม (TCP + Centralized API) ระบบที่เสนอ (GraphQL + QUIC + Rust Blockchain) ประโยชน์เชิงธุรกิจของธนาคาร
ความเร็วในการตั้งการเชื่อมต่อ ช้า (2-3 RTT จาก TCP + TLS Handshake แยกกัน) เร็วสูงสุด (0-RTT ถึง 1-RTT) เพราะควบรวมในขั้นตอนเดียว แอปพลิเคชันตอบสนองไวขึ้น ลูกค้าไม่ต้องรอนาน ลดอัตราการปิดแอปหนี
การจัดการเมื่อเครือข่ายมีปัญหา เกิดการบล็อกคอขวดที่ท่อหลัก (Head-of-Line Blocking) แยกสตรีมอิสระผ่าน QUIC หากข้อมูลคนแรกร่วง คนอื่นยังวิ่งได้ปกติ ระบบมีความเสถียรสูง (High Availability) แม้ในช่วงเวลาเร่งด่วนที่คนใช้เยอะ
การเรียกใช้ข้อมูลของหน้าบ้าน ยิงหลายรอบ (Multiple REST API Endpoints) ยิงรอบเดียวจบ (Single GraphQL Endpoint) ลดปริมาณแบนด์วิธของเซิร์ฟเวอร์ธนาคาร และทำให้เขียนโค้ดหน้าบ้านง่ายขึ้น
ความน่าเชื่อถือของข้อมูล ต้องทำ Reconciliation (กระทบยอด) สิ้นวัน แก้ไขประวัติไม่ได้ (Immutable Blockchain Ledger) ตัดต้นทุนการตรวจสอบบัญชีทุจริตและการฟอกเงิน (AML/Audit) ได้แบบ Real-time

7. ข้อควรพิจารณาเพิ่มเติมสำหรับระบบ Production (Production Readiness Checklist)

7.1 API Versioning & Backward Compatibility

  • ธนาคารแต่ละแห่งอัปเกรดระบบไม่พร้อมกัน ต้องออกแบบ API versioning (เช่น /v1/, /v2/) ตั้งแต่แรก
  • ใช้ GraphQL schema deprecation — ทำเครื่องหมาย field ที่เลิกใช้ก่อนลบจริง 1-2 release cycle
  • ต้องมี migration guide สำหรับธนาคารสมาชิกเมื่อมีการ breaking change

7.2 Audit Trail & Non-Repudiation

  • ทุกธุรกรรมต้องลงลายเซ็นดิจิทัล ED25519 และเก็บ cryptographic proof ว่าใครส่งอะไรตอนไหน
  • แม้แต่ admin หรือ operator ก็ต้องไม่สามารถแก้ไข audit log ย้อนหลังได้ — ใช้ blockchain ledger เป็น audit trail ไปในตัว
  • รองรับมาตรฐาน ISO 27001 และข้อกำหนดของธนาคารแห่งประเทศไทย (ธปท.) ด้านการตรวจสอบทางการเงิน

7.3 Rate Limiting & Circuit Breaker

  • ป้องกัน Dos / DDoS และ load spike จากธนาคารสมาชิกด้วย rate limiting แบบ per-node และ per-IP
  • ใช้ Circuit Breaker pattern (เช่น tokio-retry + futures-retry) — ถ้า Bank Node หนึ่งล่ม ต้องตัดวงจรไม่ให้ request ถาโถมซ้ำจน systemic failure
  • กำหนด backpressure strategy ชัดเจน: queue, drop, หรือ return error ทันทีเมื่อเกิน capacity

7.4 Key Management & HSM Integration

  • Private key สำหรับ sign ธุรกรรม blockchain ต้องเก็บใน Hardware Security Module (HSM) ที่รองรับ PKCS#11 หรือ cloud HSM (AWS CloudHSM, Azure Dedicated HSM)
  • ห้ามเก็บ private key ใน filesystem, environment variable, หรือ secret store ที่ไม่มี hardware root of trust
  • ออกแบบ key rotation policy — เปลี่ยน key ทุก 90 วัน โดยไม่ให้กระทบธุรกรรมที่กำลังดำเนินอยู่

7.5 Monitoring, Observability & Alerting

  • ส่ง metrics เข้า Prometheus/Grafana — วัดทั้ง request rate, latency (P50/P95/P99), error rate, และ connection pool saturation
  • ใช้ distributed tracing ด้วย OpenTelemetry — ตาม request จาก GraphQL Gateway ไปยัง Bank Node และ Blockchain Adapter
  • มี structured logging (JSON format) — ต้อง grep หา root cause ได้เร็วเวลา incident
  • ตั้ง Alerting rules: เช่น P99 latency > 500ms นานเกิน 5 นาที, error rate > 1%, หรือ connection drop spike

7.6 Rolling Upgrade & Zero-Downtime Deployment

  • Banking system ต้อง upgrade โดยไม่ปิดระบบ — รองรับ rolling update และ blue-green deployment
  • ต้องมี graceful shutdown — SIGTERM แล้วรอให้ request ค้างอยู่เสร็จก่อนค่อยปิด (tokio signal + drain timeout)
  • Connection draining — เมื่อ node กำลังจะอัปเกรด ต้องหยุดรับ request ใหม่ แต่รอให้ connection เดิมสมบูรณ์ก่อน

7.7 Data Retention & Deletion Policy (PDPA Compliance)

  • เจ้าของข้อมูลสามารถขอ ลบข้อมูล (Right to Erasure) ได้ — ต้องออกแบบ cascade deletion หรือ anonymization flow
  • ใน blockchain ที่ immutable ต้องใช้วิธี key shredding (ลบ key ที่ใช้เข้ารหัส ทิ้งให้ cipher text เปล่าประโยชน์) + เก็บ zero-knowledge proof แทน
  • กำหนด retention period ของ logs, audit trail, และ PII cache ให้ชัดเจน และมี automated cleanup job

7.8 Load Testing & SLA Benchmark

  • ไม่ใช่แค่ unit test — ต้องทำ stress test / soak test ก่อน production (ใช้เครื่องมือเช่น oha, wrk2, หรือ locust)
  • กำหนด SLA benchmark ที่วัดผลได้ เช่น P99 latency < 200ms, throughput > 10,000 tps, uptime 99.99%
  • ทดสอบ failure scenarios: ตัด network node, revoke certificate, HSM ล่ม, blockchain node lag — ระบบต้อง degrade gracefully ไม่ใช่ crash

8. ข้อควรระวังและความเสี่ยงเพิ่มเติม (Additional Risk & Considerations)

8.1 ความปลอดภัยระดับโครงสร้างพื้นฐาน (Infrastructure Security)

  • mTLS + Certificate Pinning: การเชื่อมต่อระหว่างธนาคารต้องใช้ mutual TLS ทุกฝั่ง ใบรับรองต้องมี CA hierarchy ที่เชื่อถือได้ร่วมกัน และรองรับการเพิกถอนใบรับรองทันที (CRL/OCSP stapling) หากมี node ถูกลักลอบเข้าถึง
  • Supply Chain Security: Rust crate ทุกตัวต้องผ่าน cargo vet หรือ cargo crev ก่อนนำมาใช้ใน production — banking sector เป็นเป้าหมายหลักของการโจมตีทาง supply chain (เช่น dependency confusion, typo-squatting)
  • Network Segmentation: Bank Node, Blockchain Adapter, และ GraphQL Gateway ต้องอยู่คนละ network segment กัน มี firewall policy ที่ restrict การเข้าถึงระหว่าง layer

8.2 ข้อพิจารณาด้าน Blockchain (Blockchain-Specific Concerns)

  • Consensus & Finality: ต้องเลือก consensus algorithm ที่เหมาะสม — PBFT หรือ Substrate BABE/GRANDPA? ต้องการกี่ node ถึง finalize? กรณี network partition หรือ node ล่มหมู่จะเกิด fork หรือไม่ และมี mechanism กู้คืนอัตโนมัติอย่างไร
  • Conflict Resolution: ถ้าสองธนาคารอัปเดต identity record พร้อมกัน (race condition) ใครชนะ? ต้องมี policy ชัดเจน เช่น last-writer-wins, CRDT-based merge, หรือ human-in-the-loop approval
  • Cost Projection: การรัน Substrate node 24/7 มีค่าใช้จ่ายแฝง (infra, ops, cross-datacenter bandwidth, consensus overhead) ควรเปรียบเทียบ ROI กับ centralized database + cryptographic audit trail ว่าคุ้มกับโจทย์จริงหรือไม่
  • Blockchain Node Version Drift: เมื่อเวลาผ่านไป แต่ละธนาคารอาจรัน blockchain node คนละ version — ต้องมี backward compatibility protocol และ hard fork coordination plan

8.3 Business Continuity & Disaster Recovery

  • DR / Multi-Region Failover: RPO (Recovery Point Objective) และ RTO (Recovery Time Objective) เท่าไหร่? กรณี disaster ทั้ง data center ต้องมี secondary region ที่พร้อมรับ traffic โดยไม่ต้อง rebuild blockchain node จาก cold storage
  • Personnel Security: ใช้ 4-eyes principle สำหรับทุก admin operation (เปลี่ยน config, grant/revoke certificate, sign smart contract upgrade) และมี access log ที่ audit ย้อนหลังได้แบบ tamper-proof
  • Secrets Rotation Drill: ต้องซ้อมเปลี่ยน certificate และ key เป็นระยะ เพื่อให้แน่ใจว่า DR plan ใช้ได้จริง ไม่ใช่แค่เขียนไว้ใน文档

8.4 Fraud Detection & Compliance

  • Liveness Detection: ระบบสแกนใบหน้าต้องกัน deepfake, replay attack, และ presentation attack — ใช้ passive liveness + challenge-response + AI-based spoof detection
  • Cross-Border Data Transfer: ถ้าต้องเชื่อม NDID กับต่างประเทศ (cross-border KYC) ต้องตรวจสอบข้อจำกัดทางกฎหมายของแต่ละประเทศ (เช่น GDPR ใน EU, China Data Security Law) ว่าข้อมูล PII สามารถส่งข้ามประเทศได้หรือไม่
  • AML / CTF Integration: ระบบต้องสามารถส่ง alert อัตโนมัติเมื่อ pattern การทำธุรกรรมเข้าข่ายฟอกเงิน (transaction monitoring) และเชื่อมต่อกับระบบ AMLO ของไทย

8.5 Operational Readiness

  • NTP Synchronization: Audit trail, blockchain consensus, และ event ordering จะเชื่อถือไม่ได้ถ้าเวลาในแต่ละ node ไม่ตรงกัน — ต้องใช้ dedicated NTP stratum พร้อม monitor skew ทุก 5 นาที
  • Admin Tooling: ไม่ใช่แค่ API — ต้องมี CLI และ/หรือ Web UI สำหรับ operator (เช่น ดูสถานะ node, force stop/start service, เรียกดู audit log, revoke certificate) โดยไม่ต้องแก้ database โดยตรง
  • Capacity Planning: ต้องมี baseline ของ CPU/Memory/Network/Storage usage ต่อ node เพื่อให้ scaling และ budgeting แม่นยำ — รวมถึง bottleneck analysis (เช่น IOPS ของ disk ที่ใช้รัน blockchain node)

9. คอขวด ความปลอดภัย และ Communication Mismatch ที่ต้องระวัง (Bottlenecks, Security & Protocol Gaps)

9.1 คอขวดด้านประสิทธิภาพ (Performance Bottlenecks)

  • Blockchain Consensus Latency: หากใช้ BFT consensus (3f+1 nodes) throughput จะถูกจำกัดที่ความเร็วของ consensus algorithm — Substrate BABE/GRANDPA อาจทำ ~1,000-3,000 tps แต่ถ้าเป็น banking clearing load (หลายหมื่น tx/s) blockchain จะเป็นคอขวดทันที ต้องแบ่งเป็น off-chain settlement + on-chain hash recording
  • QUIC/TCP Fallback Overshoot: หากตั้ง timeout สำหรับ QUIC fallback ต่ำเกินไป (เช่น 500ms) แต่ interbank latency ปกติอยู่ที่ 300-800ms ระบบจะ fallback ไป TCP ตลอดเวลา ทำให้ไม่เคยได้ประโยชน์จาก QUIC เลย — ต้อง tune threshold แบบ adaptive
  • GraphQL N+1 Query Problem: ถ้า resolvers แต่ละตัวยิง database แยกกันโดยไม่ใช้ DataLoader/ batch, GraphQL อาจช้ากว่า REST หลายเท่า โดยเฉพาะกรณี list identity records แล้ว resolve nested fields ทีละ record
  • HSM Signing Throughput Ceiling: HSM แต่ละรุ่นมี signing ops/sec จำกัด (เช่น ~1,000-10,000 ops/sec) หากทุก blockchain transaction ต้อง sign ด้วย HSM จะกลายเป็น bottleneck สำหรับ high-throughput
  • Legacy Core Banking IO: Core Banking ที่มีอยู่เดิมมักใช้ relational DB แบบ monolithic — under high concurrency (10k+ requests) database connection pool จะหมด และ query จะ queue ทำให้เกิด cascading latency ทั่วทั้งระบบ
  • Certificate Chain Validation on Every Handshake: mTLS + full chain validation (รวม CRL/OCSP) เพิ่ม latency ทุก connection — ต้อง implement session resumption (TLS session cache) และ QUIC 0-RTT เพื่อลด cost

9.2 ช่องโหว่ด้านความปลอดภัย (Security Blind Spots)

  • QUIC Amplification Attack: QUIC ทำงานบน UDP ซึ่ง susceptible ต่อ DDoS amplification (request เล็ก แต่ response ใหญ่) — ต้องจำกัด response size, ใช้ connection migration verification และ rate limiting ที่ edge LB
  • GraphQL Query Depth/Complexity Attack: Malicious client สามารถส่ง query ที่ทำให้ server ตีความซ้ำกันเป็น exponential (เช่น nested 10 layer) จน OOM หรือ CPU burn — ต้องตั้ง max_depth, max_complexity และ cost analysis ทุก query
  • TOCTOU (Time-of-Check to Time-of-Use): Identity verification เกิดขึ้นที่ time A แต่ actual data access เกิดขึ้นที่ time B — ข้อมูลอาจถูก revoke หรือเปลี่ยนแปลงระหว่างนั้น ต้อง validate status แบบ real-time ทุกครั้งก่อนปล่อย PII
  • Stale Certificate / CRL Propagation Delay: ถ้า node ถูกลักลอบเข้าถึงและ revoke certificate แล้ว แต่ CRL ยังไม่ propagate ไปถึง node อื่น — จะยังมีช่องว่างให้ attacker ส่ง poisoned data ได้ ต้องใช้ OCSP stapling แบบสดทุก handshake
  • Panic Safety in Rust: หากเกิด panic ใน tokio task ระหว่างที่กำลังประมวลผลธุรกรรม (โดยเฉพาะใน async resolver หรือ blockchain signing) — state leak, partial write, หรือ inconsistent data ได้ ต้องใช้ catch_unwind และ idempotency guard
  • Log Injection / Audit Tampering: หาก attacker ส่งข้อมูลที่มี control character ใส่ logs อาจทำให้ structured log parser ผิดพลาดและบดบังร่องรอยการโจมตี — ต้อง sanitize ทุก input ก่อน write log และ sign audit log ด้วย HMAC
  • Side-Channel Timing Attack on Crypto: AES-GCM และ ED25519 ใน Rust อาจ leak timing information หากไม่ได้ใช้ constant-time implementation — ต้องใช้ constant_time_eq และ audit crypto crate ทุกตัว

9.3 Communication Mismatch / Protocol Gap (การไม่สอดคล้องของโปรโตคอล)

  • Legacy Protocol Translation: Core Banking ของแต่ละธนาคารพูดภาษาโปรโตคอลไม่เหมือนกัน — บางแห่งใช้ ISO 8583 (การเงิน), บางแห่งใช้ SOAP/XML, บางแห่งใช้ IBM MQ — ต้องมี protocol adapter layer ที่แปลงเป็น GraphQL/Protobuf ที่ระบบ NDID เข้าใจ
  • Sync → Async Mismatch: Core Banking ดั้งเดิมเป็น synchronous blocking (รอผลถึงจะทำต่อ) แต่ระบบ NDID ใหม่เป็น async event-driven — ถ้า backend ยังเป็น sync อยู่ async frontend จะไม่มีประโยชน์ เพราะ最终还是 ต้องรอ blocking call อยู่ดี ต้องทำ 异步 boundary ให้ชัดเจน
  • Error Semantics Mapping: Legacy banking ส่ง error เป็น numeric code + fixed message แต่ GraphQL ส่งเป็น structured error ใน response body — ถ้าแปล semantics ไม่ตรง client จะตีความผิด เช่น "timeout" กับ "denied" ต้อง map ให้ตรงกัน
  • Identity Schema Divergence: แต่ละธนาคารมี customer data schema ไม่เหมือนกัน (บางแห่งมี 10 fields, บางแห่ง 50 fields, บางแห่งใช้ national ID บางแห่งใช้ passport) — ต้องมี canonical schema + field mapping table ที่ audit ได้ + versioned transform rule
  • Distributed Transaction Atomicity: กระบวนการ NDID หนึ่ง flow ต้องเขียนถึง Core Banking + Blockchain Adapter + P2P Tunnel ถ้าตัวใดตัวหนึ่งล้มเหลว ต้อง rollback ทั้งหมดหรือไม่? — distributed transaction โดยไม่มี coordinator ต้องออกแบบ saga pattern + compensating transaction
  • QUIC Library Interop: Rust quinn กับ QUIC implementation ของ infrastructure (Envoy, NGINX, Cloudflare) อาจ interpret spec ต่างกันเล็กน้อย — ต้อง interop test ทุกคู่ก่อน deploy จริง
  • Load Balancer Blind Spot: LB รุ่นเก่า (NGINX 1.x, HAProxy 1.x) ไม่ support HTTP/3 หรือ UDP health check — ถ้าใช้ QUIC แต่ LB ตรวจ health node ไม่ได้ จะเกิด routing ไป node ที่ dead ได้ ต้อง upgrade LB รุ่นที่รองรับ HTTP/3 หรือใช้ L4 UDP proxy

10. แผนการทดสอบและการส่งมอบ (Deployment & Testing Steps)

  1. สัปดาห์ที่ 1-4 (Sandbox Isolation): พัฒนาตัวโปรโตไทป์โครงสร้างหลักด้วย Rust และเปิดใช้งานระบบจำลอง Blockchain Node ขึ้นมาในสถาปัตยกรรมปิดเพื่อทดสอบค่าขีดจำกัดความเร็ว
  2. สัปดาห์ที่ 5-8 (Network Hardening & Firewall Tuning): นำระบบเครือข่าย QUIC เข้าไปคุยกับฝ่าย Network Security ของธนาคารเพื่อเซ็ตค่าอัปเดตตัว Load Balancer (เช่น Envoy/NGINX รองรับ HTTP/3) และตั้งค่ายอมรับพอร์ต UDP ภายใต้เงื่อนไขความปลอดภัย
  3. สัปดาห์ที่ 9-12 (GraphQL Integration & Fallback Drill): ต่อพ่วงแอปพลิเคชันหน้าบ้านเข้ากับคลังข้อมูล GraphQL Gateway ทำการทดลองสุ่มดึงปลั๊กเครือข่าย UDP ออกเพื่อทดสอบว่าระบบสลับไปใช้ TCP ได้อย่างนุ่มนวลและไม่ส่งผลเสียต่อธุรกรรมของลูกค้า
